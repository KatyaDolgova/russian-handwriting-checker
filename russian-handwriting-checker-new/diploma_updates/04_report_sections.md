РАЗДЕЛ 3.3 — ДИАГРАММА КЛАССОВ
Блок описания модели Check — ЗАМЕНИТЬ
================================================================================

Модель Check представляет результат одной проверки и содержит следующие
атрибуты: идентификатор записи, идентификаторы пользователя, ученика, функции
и папки (все внешние ключи), имя загруженного файла, название работы (title),
исходный текст задания (source_text — для диктантов и изложений, где требуется
сравнение с эталоном), текст ученика (original_text), исправленный текст
(corrected_text), массив ошибок (errors, тип JSON), баллы по критериям
(criteria, тип JSONB — объект вида {«К1»: {score, max, comment}}), результат
зачёт/незачёт (pass_fail, тип TEXT), числовая оценка (score), максимальный
балл (score_max), текстовый комментарий, дата работы (work_date) и дата
создания записи (created_at).

Поля criteria и pass_fail являются взаимоисключающими: при наличии pass_fail
запись исключается из расчёта числовой статистики и отображается отдельным
счётчиком зачётов в интерфейсе.

--- Блок описания модели Function — ЗАМЕНИТЬ ---

Модель Function хранит настраиваемые функции проверки. Помимо идентификатора,
имени, описания, системного промпта и шаблона пользователя, модель содержит
два дополнительных числовых параметра: score_max (максимальный балл,
используется при отсутствии информации из поля criteria) и min_words
(минимальное количество слов, необходимое для работы данного типа). Флаги
is_default и is_published управляют доступностью функции.

--- Блок описания FunctionRepository — ДОПОЛНИТЬ ---

Добавить в конец описания:

Метод create_version(function_id, change_note) создаёт снапшот текущего
состояния функции как новую запись в таблице function_versions с автоматически
назначаемым номером версии. Это позволяет учителю вручную зафиксировать
значимое изменение, не публикуя функцию в галерею. Метод delete_version
(version_id) удаляет конкретную версию; текущая функция при этом не изменяется.

РАЗДЕЛ 4.2 — СЕРВЕРНАЯ ЧАСТЬ: КЛЮЧЕВЫЕ КОМПОНЕНТЫ

--- Добавить абзац про _parse() после описания метода ---

Метод _parse() выполняет разбор ответа языковой модели. Перед поиском
JSON-блока метод очищает возможную Markdown-обёртку (```json ... ```),
которую модель иногда добавляет к ответу вопреки инструкции. Очистка
выполняется регулярными выражениями:

    cleaned = re.sub(r'^```(?:json)?\s*', '', raw.strip())
    cleaned = re.sub(r'\s*```\s*$', '', cleaned)

После очистки метод ищет JSON-блок через re.search(r'\{[\s\S]*\}', cleaned).
При ошибке десериализации выполняется попытка исправить типичную ошибку —
лишние запятые перед закрывающей скобкой. Параметр max_tokens для
LLMService увеличен до 8000 для корректной обработки больших текстов
с множеством ошибок.

--- Добавить абзац про _build_result() ---

Метод _build_result() формирует итоговый словарь результата из разобранного
JSON. Помимо ранее описанных полей (corrected_text, errors, score,
html_highlighted, comment), метод теперь передаёт:
- criteria — словарь баллов по критериям, полученный из поля «criteria» ответа LLM;
- pass_fail — результат «зачёт»/«незачёт», если модель вернула нечисловую оценку;
- score_max — максимальный балл из настроек функции (fn_score_max);
- min_words — минимальное количество слов из настроек функции (fn_min_words);
- score_label — строковое представление оценки, если она не является числом.
Флаг is_generation устанавливается в True, когда ответ не содержит полей
«corrected» и «errors» — то есть модель сгенерировала произвольный текст
вместо структурированного JSON.

РАЗДЕЛ 4.2 — СЕРВЕРНАЯ ЧАСТЬ: КЛЮЧЕВЫЕ КОМПОНЕНТЫ
Подраздел «Функции проверки» — ДОПОЛНИТЬ блок про версионирование
================================================================================

--- Заменить абзац про версионирование ---

БЫЛО:
При редактировании уже опубликованной функции автоматически создаётся новая
версия в таблице function_versions.

СТАЛО:
Система поддерживает два режима версионирования. Автоматический режим: при
сохранении изменений в уже опубликованной функции маршрут PUT /functions/{id}
сохраняет значение флага is_published до вызова repo.update() и после
завершения обновления вызывает repo.republish(), который создаёт новую запись
в function_versions с текущим состоянием промпта. Ручной режим: учитель может
в любой момент зафиксировать снапшот функции через POST /functions/{id}/versions
с необязательным параметром change_note — это позволяет сохранять версии
черновиков, не публикуя их в галерею. Версии можно удалять через
DELETE /functions/{id}/versions/{ver_id}. История версий опубликованных функций
доступна для просмотра другим пользователям (только чтение): маршрут
GET /functions/{id}/versions проверяет либо владельца, либо флаг is_published.

--- Добавить новый подраздел в 4.2 ---

4.2.X Поддержка кириллических имён файлов в OCR

Функция cv2.imread() библиотеки OpenCV не поддерживает пути с символами
кириллицы на платформе Windows из-за ограничений Windows API. Для решения
этой проблемы в модуле image_processing.py реализована вспомогательная
функция _imread_unicode(), которая читает файл как байтовый поток через
numpy.fromfile() и передаёт буфер в cv2.imdecode():

    def _imread_unicode(image_path: str) -> np.ndarray:
        buf = np.fromfile(image_path, dtype=np.uint8)
        img = cv2.imdecode(buf, cv2.IMREAD_COLOR)
        if img is None:
            raise ValueError(f"Не удалось загрузить изображение: {image_path}")
        return img

Функции enhance_for_paddle() и enhance_for_tesseract() используют
_imread_unicode() вместо прямого вызова cv2.imread().

РАЗДЕЛ 4.3 — КЛИЕНТСКАЯ ЧАСТЬ: КЛЮЧЕВЫЕ КОМПОНЕНТЫ
Добавить новый компонент HighlightedText
================================================================================

--- Добавить после описания компонента CheckMini ---

Компонент HighlightedText реализует интерактивную подсветку ошибок в
сохранённом тексте проверки. Компонент принимает исходный текст и массив
ошибок, разбивает строку на сегменты по вхождениям поля original каждой
ошибки и рендерит каждый ошибочный сегмент как отдельный span с волнистым
красным подчёркиванием (Tailwind-класс decoration-wavy decoration-red-400).
При наведении курсора появляется тултип с тремя полями: тип ошибки
(орфография / пунктуация / грамматика / стиль), исправленный вариант (corrected)
и краткое пояснение (comment). Тултип реализован через React-состояние
(onMouseEnter / onMouseLeave) без внешних библиотек, имеет ширину 256 px
и поддерживает перенос длинного текста. Компонент используется в панели
истории проверок (HistoryPanel) для отображения исходного текста работы.

РАЗДЕЛ 4.3 — КЛИЕНТСКАЯ ЧАСТЬ
Обновить описание компонента ResultPanel (зачёт/незачёт)
================================================================================

--- Добавить абзац в описание ResultPanel ---

Для функций с оценкой «зачёт»/«незачёт» (итоговое сочинение) компонент
определяет тип результата по полю pass_fail или по наличию поля result
(вместо score) в каждом критерии. При обнаружении такого результата вместо
числовых полей ввода оценки отображается выпадающий список с вариантами
«Зачёт» и «Незачёт». При сохранении в API передаётся поле pass_fail вместо
score. В панели истории и на странице учеников записи с pass_fail отображаются
отдельно: зелёный или красный бейдж «З» / «Н» вместо числового процента;
в статистике выводится счётчик «X/Y зачётов» и не включается в расчёт
среднего процента успеваемости.

ПРИЛОЖЕНИЕ В — СХЕМА БАЗЫ ДАННЫХ
Описание для подписи к ER-диаграмме
================================================================================

Схема базы данных содержит 8 таблиц. Центральной является таблица checks,
связанная с таблицами pupils (многие-к-одному), folders (многие-к-одному)
и users через поле user_id. Таблица functions хранит настраиваемые функции
проверки; связанная с ней таблица function_versions обеспечивает историю
версий каждой функции. Связь между учениками и группами реализована через
промежуточную таблицу pupil_groups (отношение многие-ко-многим). Профиль
учителя хранится в таблице user_profiles.

Таблица checks содержит поля criteria (JSONB) для хранения баллов по
критериям ОГЭ/ЕГЭ и pass_fail (TEXT) для результата итогового сочинения.
Эти поля являются взаимоисключающими: запись имеет либо числовую оценку
с детализацией по критериям, либо результат зачёт/незачёт.
